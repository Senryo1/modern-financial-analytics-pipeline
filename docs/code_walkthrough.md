# Code Walkthrough

## Project Name

Modern Financial Analytics Pipeline

## Purpose

This document explains the design logic behind the main Python scripts used in the project.

The goal is not only to describe what each script does, but also to explain why the pipeline is structured this way and how it could evolve into a more production-oriented data workflow.

---

## 1. Pipeline Overview

The project follows a simple but extensible data workflow:

```text
Sample Data Generation
        ↓
Raw Data Layer
        ↓
Transformation Layer
        ↓
Processed Data Layer
        ↓
Validation Layer
        ↓
SQL Data Quality and Analytics Layer
        ↓
Reporting / Dashboard Layer
```

The current Python scripts represent the first three operational stages of the pipeline:

| Script | Main Responsibility |
|---|---|
| `generate_sample_data.py` | Create reproducible simulated financial datasets |
| `transform.py` | Clean and standardize raw datasets |
| `validate.py` | Run data quality and business rule validations |

The SQL scripts then provide database-oriented validation and analytical logic.

---

## 2. Design Philosophy

The code was intentionally separated into multiple scripts instead of being written as one large file.

This separation follows a basic data pipeline design principle:

```text
Each layer should have one main responsibility.
```

The design separates:

- data generation;
- data transformation;
- data validation;
- SQL quality checks;
- SQL analytics;
- documentation.

This makes the project easier to understand, maintain, test, and extend.

---

## 3. `generate_sample_data.py`

## Purpose

The `generate_sample_data.py` script creates simulated datasets for a financial or microfinance portfolio.

Generated datasets:

```text
customers.csv
branches.csv
advisors.csv
products.csv
loans.csv
payments.csv
```

The script writes full generated data to:

```text
data/raw/
```

It also writes smaller portfolio-review samples to:

```text
data/sample/
```

---

## Main Design Decisions

### 3.1 Reproducible sample generation

The script uses a fixed random seed:

```python
random.seed(42)
```

This ensures the generated data remains consistent across executions.

That matters because reproducibility is important in data projects. If someone else runs the script, they should get the same general output and be able to reproduce the pipeline behavior.

---

### 3.2 Raw data is generated, not manually uploaded

The raw datasets are generated by code instead of being manually stored in GitHub.

This is intentional.

The `data/raw/` folder is ignored through `.gitignore`, while the script remains versioned.

This means the project stores the logic to generate data, not unnecessary generated outputs.

This keeps the repository cleaner and avoids version noise.

---

### 3.3 Sample data is included for portfolio review

The script also creates smaller sample files under:

```text
data/sample/
```

These files are versioned in GitHub so that reviewers can inspect the structure of the data without running the script.

This balances two goals:

- avoid versioning full raw/generated files;
- still provide visible evidence of the data model.

---

### 3.4 Dataset relationships are built into the generation logic

The script does not generate isolated flat files. It creates related datasets.

Examples:

- loans reference customers;
- loans reference products;
- loans reference branches;
- loans reference advisors;
- payments reference loans;
- advisors reference branches.

This matters because the project is designed to simulate a relational financial data environment.

That allows later validations around referential integrity and business consistency.

---

## Key Functions

### `random_date()`

Generates a random date between two dates.

This supports more realistic simulation of:

- birth dates;
- registration dates;
- hire dates;
- disbursement dates;
- payment dates.

---

### `save_csv()`

Centralizes the logic for saving full datasets to `data/raw/`.

This avoids repeating output-path logic for every dataset.

---

### `save_sample_csv()`

Centralizes the logic for saving small sample datasets to `data/sample/`.

This supports portfolio review without versioning the complete generated raw files.

---

### Dataset generation functions

Each dataset has its own generation function:

```text
generate_branches()
generate_advisors()
generate_products()
generate_customers()

```

This makes the script modular.

If the data model grows later, new datasets can be added without rewriting the entire file.

## 4. `transform.py`

## Purpose

The `transform.py` script reads generated raw datasets from:

```text
data/raw/
```

and writes cleaned, standardized datasets to:

```text
data/processed/
```

The transformation layer is responsible for preparing data before quality checks and analysis.

---

## Main Design Decisions

### 4.1 Transformation is separated from generation

The project does not clean data inside `generate_sample_data.py`.

This is intentional.

Even though the data is generated by us, the project simulates a real-world workflow where raw source data arrives first and is transformed later.

The separation creates a clearer data lineage:

```text
raw source data → transformed processed data
```

---

### 4.2 Common transformations are reusable

The script includes generic helper functions such as:

```text
standardize_column_names()
clean_text_columns()
convert_date_columns()
convert_numeric_columns()
```

These functions are not tied to one dataset. They can be reused across customers, loans, payments, products, branches, and advisors.

This design reduces duplication and makes the transformation logic easier to maintain.

---

### 4.3 Dataset-specific transformation functions

The script also includes dataset-specific functions:

```text
transform_customers()
transform_branches()
transform_advisors()
transform_products()
transform_loans()
transform_payments()
```

This creates a useful separation:

| Layer | Purpose |
|---|---|
| Generic helper functions | Apply reusable transformation logic |
| Dataset-specific functions | Define which transformations apply to each dataset |

For example, not every dataset has date columns or numeric columns. Dataset-specific functions allow each table to apply only the relevant transformations.

---

### 4.4 Processed data is generated output

The `data/processed/` folder is also ignored by Git.

This is appropriate because processed files are generated outputs. They can be recreated by running the transformation script after raw data is generated.

The repository stores the transformation logic, not the output noise.

---

## Key Functions

### `read_csv()`

Reads a raw CSV file from the raw data directory.

It also checks whether the expected file exists before attempting to read it.

This makes failures more explicit.

---

### `save_processed_csv()`

Writes a processed dataset to `data/processed/`.

Centralizing this logic avoids repeated path handling and keeps output behavior consistent.

---

### `standardize_column_names()`

Normalizes column names using lowercase and snake_case formatting.

This improves consistency across the pipeline and makes SQL or downstream modeling easier.

---

### `clean_text_columns()`

Trims unnecessary whitespace from text fields.

This prevents common data quality issues, especially in categorical fields and identifiers.

---

### `convert_date_columns()`

Converts selected fields into date-compatible formats.

Using explicit date conversion helps prevent downstream analytical issues in time-based reporting.

---

### `convert_numeric_columns()`

Converts selected fields into numeric formats.

This supports financial calculations, aggregations, and validation checks.

## 5. `validate.py`

## Purpose

The `validate.py` script performs data quality validations on the processed datasets.

It is designed to evaluate whether the data is ready for analysis and reporting.

Input folder:

```text
data/processed/
```

Output:

```text
Validation results printed in the terminal
```

---

## Main Design Decisions

### 5.1 Validation is separated from transformation

The script does not modify the data.

Its role is to evaluate whether the processed datasets comply with expected technical and business rules.

This keeps the pipeline conceptually clean:

```text
transform.py = changes the data
validate.py  = evaluates the data
```

This separation is important because transformation and validation are different responsibilities.

---

### 5.2 Generic validation functions

The script defines reusable validation functions such as:

```text
check_missing_values()
check_duplicates()
check_allowed_values()
check_positive_values()
check_non_negative_values()
check_referential_integrity()
```

These functions can be applied to different datasets and columns.

This design avoids hardcoding every check manually and supports future extension.

---

### 5.3 Business rule validation functions

The script also includes business-specific validations such as:

```text
check_payment_dates_after_disbursement()
check_missed_payments_have_zero_amount()
check_paid_payments_have_zero_days_late()
```

These are not just technical validations. They represent business logic.

This is important because data quality is not only about format or completeness. In financial analytics, data quality also depends on whether records make sense in business terms.

---

## Validation Categories

The script currently validates:

| Category | Example |
|---|---|
| Missing values | Required fields should not be null |
| Duplicates | Primary keys should be unique |
| Allowed values | Status fields must use expected categories |
| Numeric rules | Loan amounts must be positive |
| Referential integrity | Payments must reference valid loans |
| Business consistency | Payments cannot occur before disbursement |

---

## Example: Referential Integrity

The validation function:

```text
check_referential_integrity()
```

is designed to verify that child table keys exist in parent tables.

Example:

```text
payments.loan_id → loans.loan_id
```

This means that every payment must be linked to a valid loan.

This is a critical control in financial data environments because orphan records can distort reporting, recovery analysis, and customer-level views.

---

## Example: Business Rule Validation

The rule:

```text
payment_date >= disbursement_date
```

ensures that a payment does not occur before a loan was issued.

This type of validation connects data engineering with business logic.

It demonstrates that the pipeline is not only checking structure, but also the plausibility of financial events.

## 6. SQL Layer

The project also includes two SQL files:

```text
sql/data_quality_checks.sql
sql/analytics_queries.sql
```

These files complement the Python pipeline.

---

## `sql/data_quality_checks.sql`

This file contains SQL-based validations.

The logic is:

```text
If a query returns rows, those rows represent records that need review.
If a query returns zero rows, the validation passes.
```

This is a common pattern in SQL-based data quality checks.

The SQL validation layer demonstrates how similar controls could be implemented directly in:

- SQL Server;
- PostgreSQL;
- Snowflake;
- a cloud data warehouse;
- dbt models and tests.

---

## `sql/analytics_queries.sql`

This file contains business-oriented analytical queries.

It includes:

- executive portfolio KPIs;
- branch performance;
- advisor performance;
- product performance;
- customer segment analysis;
- monthly trends;
- recovery analysis;
- Power BI-ready analytical datasets.

This layer translates validated data into reporting and decision-support outputs.

## 7. Current Limitations

The current version is intentionally simple and portfolio-oriented.

Main limitations:

1. Data is generated locally, not ingested from real source systems.
2. There is no database loading layer yet.
3. SQL scripts are documentation/executable logic but are not currently automated against a database.
4. Validation results are printed to the terminal instead of saved to structured reports.
5. There is no orchestration layer yet.
6. There is no formal testing framework such as `pytest`.
7. There is no dbt project yet.
8. There is no cloud warehouse layer yet.
9. There is no Power BI dashboard yet.

These limitations are acceptable at the current stage because the project is being built incrementally.
